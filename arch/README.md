# 废弃项目存档 / Archived Projects

## latextab_in_html 计划（已废弃）

> **归档日期**: 2026-04-08  
> **归档原因**: 时间不足 + 技术力不足 + 边际收益递减

---

### 项目背景与动机

在准备 CVPR Poster 和学术展示材料时，产生了一个想法：**能否在网页中完美复刻 LaTeX 编译生成的 PDF 表格效果？**

动机源自实际痛点：
- LaTeX 生成的 PDF 表格美观、专业，符合 IEEE/CVPR 学术规范
- 但网页展示需要更灵活的交互和响应式适配
- 希望找到一种"既能保持 LaTeX 学术美感，又能享受 Web 技术便利"的方案

---

### 完整探索历程

#### 阶段一：需求明确（思维转变的起点）

最初的需求模糊："让网页表格看起来像 LaTeX"。经过反复迭代，逐渐明确为具体的技术指标：

| 要素 | LaTeX PDF 特征 | 网页实现难点 |
|------|---------------|-------------|
| **字体** | Latin Modern / Times New Roman 衬线字体 | Web 字体加载、跨平台一致性 |
| **三线表** | \toprule(2pt)、\midrule(1pt)、\bottomrule(2pt) | CSS border 精度、高清屏渲染差异 |
| **数字对齐** | 小数点对齐、等宽数字 | HTML table 单元格宽度计算复杂 |
| **间距** | 紧凑的 \small / \footnotesize 行距 | CSS line-height 与 padding 的微妙平衡 |
| **分组线** | \cmidrule 部分横线分隔方法组 | 需要精确控制 tr 的 border-bottom |

#### 阶段二：技术方案对比（8 种实现）

最终实现了 **8 种不同的技术方案对比页面**，每种都尝试用不同技术栈还原 LaTeX 效果：

1. **HTML + Tailwind CSS**  
   - 纯手写 CSS，手动控制 border、padding、font-family
   - 最接近日常开发方式，但微调耗时

2. **MathJax + LaTeX (Booktabs)**  
   - 在网页中直接嵌入 LaTeX 源码，由 MathJax 渲染
   - 理论上最"正宗"，但实际渲染效果与 PDF 仍有差距

3. **LaTeX.css**  
   - 使用开源项目 latex-css 的样式
   - 依赖外部库，定制化受限

4. **Tufte CSS**  
   - 模仿 Edward Tufte 的信息设计风格
   - 过于极简，不适合数据密集型表格

5. **Custom Booktabs CSS**  
   - 专门定制的 Booktabs 风格 CSS
   - 线宽层次丰富，但跨浏览器一致性差

6. **Quarto**  
   - 数据科学领域的现代出版工具风格
   - 带有背景色和圆角，偏离纯 LaTeX 风格

7. **Tailwind CSS + Booktabs**  
   - 结合 Tailwind 工具类与 Booktabs 逻辑
   - 在灵活性与规范性之间折中

8. **MDX + Tailwind Typography**  
   - 模拟 Astro 生态中的 MDX 组件映射方案
   - 最接近生产环境实践，但复杂度最高

#### 阶段三：迭代与优化（死亡螺旋）

在开发过程中，陷入了**细节优化的无限循环**：

- **字体迭代**: Times New Roman → Latin Modern Roman → Literata (Google Fonts 近似)
- **线宽迭代**: px → pt → 不同浏览器的渲染差异
- **间距迭代**: 反复调整 padding、line-height、vertical-align
- **对齐迭代**: 左对齐、居中、小数点对齐的各种尝试

每次调整都需要刷新页面、对比 PDF、再微调，**时间成本呈指数增长**。

---

### 废弃原因（深层分析）

#### 1. 时间不足 — 边际收益递减的经典案例

- **初期**: 快速搭建 8 种方案，成就感高
- **中期**: 微调字体和线宽，耗时但可见改进
- **后期**: 追求"像素级完美"，投入数小时只能改进 1% 的效果

**关键认知**: 对于学术展示，"足够好"与"完美"之间的差距，观众几乎无法察觉，但开发者要付出 10 倍时间。

#### 2. 技术力不足 — 媒介本质差异

LaTeX 与 HTML 是根本不同的媒介：

| 维度 | LaTeX | HTML/CSS |
|------|-------|----------|
| 渲染引擎 | TeX 排版引擎（专用算法） | 浏览器引擎（通用布局） |
| 字体处理 | 嵌入字形轮廓，精确控制 | 依赖系统字体 + Web Fonts |
| 断行断页 | 算法优化，全局最优 | 响应式，流式布局 |
| 精度 | 物理尺寸（pt/mm） | 逻辑像素，DPI 差异 |

**核心问题**: HTML 不是为"精确排版"设计的，强行模拟 TeX 的排版算法是**逆媒介特性**的行为。

#### 3. 工程化困境 — 维护成本考量

即使投入时间做出"80% 像"的效果，后续维护也是问题：
- 浏览器更新可能破坏精确调整的样式
- 不同设备（Retina vs 非 Retina）渲染差异
- 表格内容变化时需要重新调整所有列宽

### 思维转变（关键教训）

> **从 "在网页中复刻 LaTeX" 转向 "为不同媒介选择最适合的工具"**

**错误的思维模式**:
- 单一源文件，多格式输出
- 网页应该看起来像 PDF
- 追求像素级一致

**正确的思维模式**:
- **PDF 场景**: 直接用 LaTeX 编译，保证学术规范
- **网页场景**: 使用适合 Web 的表格设计（Flowbite/Preline 等现代组件）
- **打印场景**: 专门的打印样式 @media print

每种媒介有其**最佳实践**，强行统一只会导致所有场景都是次优解。

---

### 替代方案建议

1. **直接使用 LaTeX PDF**
   - 对于正式学术场合（论文、海报），PDF 仍是金标准
   - 无需妥协，完美排版

2. **Quarto / Typst 生成 HTML**
   - 如果使用 Markdown 源文件，Quarto 可以生成高质量的 HTML 表格
   - 专业团队维护，跨平台一致性好

3. **接受网页的局限性**
   - 网页表格追求"清晰可读"而非"像素级复刻"
   - 使用成熟的 UI 组件库（Flowbite、Preline、shadcn/ui）

### 经验总结

**对于类似的技术探索**:

1. **设定止损点**: 在开始前明确 "最多投入 X 小时，如果无法达到 Y 效果就放弃"
2. **接受媒介特性**: 不同技术有其设计目标，强行跨界往往事倍功半
3. **区分"演示"与"生产"**: 概念验证（PoC）不需要工程级完美
4. **及时归档**: 失败的项目也有价值，记录教训避免重复踩坑

> **技术选型原则**: 选择**最适合当前场景**的工具，而非**理论上最强大**的工具。
